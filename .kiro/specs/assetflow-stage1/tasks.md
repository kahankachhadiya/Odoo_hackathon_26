# Implementation Plan: AssetFlow Stage 1

## Overview

Implement the foundational infrastructure for AssetFlow: Supabase database schema (tables, enums, trigger, RLS), a TypeScript Supabase client with generated types, authentication service, role-based routing with an Admin route guard, and the Organization Setup admin panel (Departments, Categories, Employee Directory tabs).

---

### Naming Conventions

| Item | Convention | Example |
|------|-----------|---------|
| React components | PascalCase | `AdminGuard.tsx`, `DepartmentsTab.tsx` |
| Services & utilities | camelCase | `authService.ts`, `validation.ts` |
| Type definitions | camelCase | `index.ts` in `types/` |
| Test files | `*.test.ts` / `*.test.tsx` | `validation.test.ts` |
| Property test files | `*.property.ts` | `nameValidation.property.ts` |
| SQL migration files | `migration_YYYYMMDD_<description>.sql` | `migration_20260712_add_email_to_profiles.sql` |
| Supabase generated types | `database.types.ts` | do not rename or hand-edit |

## Notes

- Tasks marked with `*` are optional and can be skipped for a faster MVP
- Property tests use [fast-check](https://github.com/dubzzz/fast-check) with a minimum of 100 iterations per property
- Integration tests require a connection to the cloud Supabase project (set `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` in `.env.local`); there is no local Supabase instance
- Authentication uses Google OAuth via Supabase. Enable the Google provider in the Supabase Dashboard → Authentication → Providers before testing the auth flow
- The OAuth callback redirect URL (e.g., `http://localhost:5173/auth/callback`) must be added to the allowed redirect URLs in the Supabase Dashboard → Authentication → URL Configuration
- The `is_admin()` SECURITY DEFINER function must be created in `schema.sql` before any RLS policy that calls it
- `departments.head_id` FK is added via `ALTER TABLE` after `profiles` is created — maintain this ordering in `schema.sql`
- Each property test file should include the annotation comment: `// Feature: assetflow-stage1, Property {N}: {property_text}`
- All requirements references use the granular sub-requirement format (e.g., "Requirements 6.2, 6.3")
